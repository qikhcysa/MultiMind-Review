## 1. 项目概述

**MultiMind-Review** 是一个面向电商场景的智能商品评论分析系统。它能够：

- **识别评论对应的商品**（从知识库向量匹配）
- **检测评论涉及的多个维度**（如质量、物流、价格、服务等）
- **为每个维度打情感评分**（1–5分，标注正/中/负情感）
- **支持多轮对话式分析**（ReAct 编排器，动态调用工具，跟进追问）
- **对大批量评论进行聚类**，自动提炼每个簇的核心主题
- **全程记录审计日志**，追溯每一步 AI 的决策过程
- **提供无代码配置界面**，支持动态增删商品/维度/参数

## 2. 整体架构设计

系统采用 **多智能体（Multi-Agent）+ 检索增强生成（RAG）** 的架构，共有两种使用方式：

**方式一：固定流水线（ReviewAnalysisPipeline）**  
四个专职智能体按固定三阶段顺序协作，适合批量结构化分析：

```
用户评论文本
      │
      ▼
┌─────────────────────────────┐
│  Stage 1: 商品识别           │
│  ProductRecognitionAgent    │  ← 向量相似度匹配产品知识库
└──────────────┬──────────────┘
               │ ProductMatch | None
               ▼
┌─────────────────────────────┐
│  Stage 2: 维度检测           │
│  DimensionDetectionAgent    │  ← LLM 提取 / 关键词兜底
└──────────────┬──────────────┘
               │ list[dimension_id]
               ▼
┌─────────────────────────────┐
│  Stage 3a: 证据检索          │
│  EvidenceRetrievalAgent     │  ← 句子级 RAG，按维度检索证据片段
└──────────────┬──────────────┘
               │ {dimension_id: [evidence_sentences]}
               ▼
┌─────────────────────────────┐
│  Stage 3b: 情感评分          │
│  SentimentScoringAgent      │  ← LLM 对每维度打分 / 关键词兜底
└──────────────┬──────────────┘
               │ list[DimensionScore]
               ▼
      汇总 & 输出 ReviewAnalysisResult
      + 写入 AuditTrail
```

**方式二：ReAct 对话编排（OrchestratorAgent）**  
在流水线之上新增一层 ReAct 风格的编排器，让 LLM 动态决定调用哪个工具、以何种顺序调用，并支持多轮追问：

```
用户消息（评论 或 追问）
      │
      ▼
┌──────────────────────────────────────┐
│        OrchestratorAgent             │
│  (ReAct Loop: Reason → Act → Observe)│
│                                      │
│  LLM ──function_calling──► TOOLS     │
│       recognize_product              │
│       detect_dimensions   ────────►  │ 调用各专职 Agent
│       retrieve_evidence              │
│       score_sentiment                │
│                                      │
│  多轮 _history 维持对话上下文          │
│  max_iterations 防止无限循环           │
└──────────────────────────────────────┘
      │
      ▼
  中文自然语言回复（含维度评分 + 商品识别 + 整体评价）
```

## 5. 从零搭建：逐层拆解

### 5.1 数据模型层（Pydantic）

**文件**：`src/models/__init__.py`

所有在系统中流动的数据都使用 **Pydantic v2** 定义为强类型模型，确保数据的正确性和一致性。

#### 核心模型

```python
class ProductInfo(BaseModel):
    id: str                      # 商品唯一标识
    name: str                    # 商品名称
    brand: str                   # 品牌
    category: str                # 类别
    description: str             # 详细描述
    features: list[str]          # 特性列表
    keywords: list[str]          # 检索关键词

class Dimension(BaseModel):
    id: str                      # 维度ID（如 "quality"）
    name: str                    # 中文名（如 "产品质量"）
    name_en: str                 # 英文名
    description: str             # 维度描述
    keywords: list[str]          # 关键词列表

class DimensionScore(BaseModel):
    dimension_id: str
    score: float                 # 1.0 – 5.0（Pydantic 校验 ge=1.0, le=5.0）
    sentiment: str               # "positive" | "neutral" | "negative"
    evidence: list[str]          # 来自评论的原文证据片段
    reasoning: str               # LLM 的评分说明

class ReviewAnalysisResult(BaseModel):
    review_id: str
    review_text: str
    product_match: ProductMatch | None
    detected_dimensions: list[str]
    dimension_scores: list[DimensionScore]
    overall_sentiment: str
    overall_score: float | None
    analysis_timestamp: datetime
```

**设计要点**：
- `DimensionScore.score` 使用 `ge=1.0, le=5.0` 约束，防止 LLM 输出越界值
- `AuditEntry` 模型记录每个智能体的输入/输出/检索结果/推理过程，实现完整可追溯性
- `ClusterResult` 包含 `noise_count`，明确标识 DBSCAN 未归类点数

### 5.3 RAG 检索增强层（Embeddings + ChromaDB）

**文件**：`src/rag/embeddings.py`、`src/rag/vector_store.py`

这是整个系统的**信息检索基础设施**，为商品识别和证据检索提供向量相似度能力。

#### 嵌入模型（`embeddings.py`）

```python
class EmbeddingModel:
    def encode(self, texts: list[str], normalize: bool = True) -> np.ndarray:
        ...
    def encode_single(self, text: str) -> np.ndarray:
        ...
```

- **主路径**：加载 `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`，输出 384 维向量
- **离线兜底**：若 HuggingFace 不可访问，自动切换到基于 SHA256 哈希的确定性伪嵌入（128 维）
- **归一化**：默认 L2 归一化，使向量点积等价于余弦相似度

#### 向量存储（`vector_store.py`）

封装 ChromaDB 的核心操作：

```python
class VectorStore:
    def upsert(self, collection_name, ids, documents, embeddings, metadatas):
        """存储向量（幂等，重复写入会更新）"""

    def query(self, collection_name, query_embedding, top_k) -> dict:
        """向量相似度检索，返回 top-k 邻居及相似度分数"""
        # similarity = 1 - cosine_distance

    def delete_collection(self, collection_name):
        """删除集合（用于清理临时评论集合）"""
```

- 持久化路径由 `CHROMA_PERSIST_DIR` 环境变量控制（默认 `./chroma_db`）
- 使用余弦距离作为相似度度量
- 延迟初始化（首次调用时才创建 ChromaDB 客户端）

### 5.4 多智能体层（Multi-Agent）

**文件**：`src/agents/`

系统包含 4 个专职智能体，每个智能体职责单一、输入输出明确，并且**每个智能体都有 LLM 路径和关键词/启发式兜底路径**，确保在无 API Key 的环境下也能运行。

#### 智能体一：ProductRecognitionAgent（商品识别）

```
输入：评论文本
输出：ProductMatch | None，AuditEntry
```

**工作原理**：
1. **建库阶段**（`index_products`）：将每个商品的名称、品牌、类别、描述、特性、关键词拼接为文档，用嵌入模型向量化后存入 ChromaDB `products` 集合
2. **识别阶段**（`recognize`）：对评论文本生成嵌入，在向量库中检索 Top-5 最相似商品
3. **决策逻辑**：若最高相似度 ≥ 阈值（默认 0.5），返回匹配结果；否则返回 `None`

```python
# 商品文档构造方式
doc = f"{p.name} | {p.brand} | {p.category} | {p.description} | {' '.join(p.features)} | {' '.join(p.keywords)}"
```

---

#### 智能体二：DimensionDetectionAgent（维度检测）

```
输入：评论文本
输出：list[detected_dimension_ids]，AuditEntry
```

**LLM 路径**：
```
System Prompt：你是商品评论分析专家，从评论中识别涉及哪些评价维度
User Prompt：[维度列表] + [评论文本]
LLM 返回：{"detected_dimensions": ["quality", "delivery"], "reasoning": "..."}
```

**关键词兜底**：遍历所有维度的 `keywords` 字段，在评论文本中匹配，有命中则认为涉及该维度。

---

#### 智能体三：EvidenceRetrievalAgent（证据检索）

```
输入：评论文本、维度列表
输出：{dimension_id: [evidence_sentence, ...]}, list[AuditEntry]
```

**工作原理**（这是最精妙的设计之一）：
1. **句子切分**：按 `。！？；\n.!?;` 将评论切成若干句子
2. **临时向量库**：为当前评论创建 `review_{review_id}` 临时集合，将所有句子向量化存入
3. **按维度检索**：用 `{维度名} {维度描述}` 作为查询，检索 Top-K 相关句子作为该维度的证据
4. **清理**：分析完成后删除临时集合，不污染持久化存储

这实现了**句子级 RAG**：不是把整条评论喂给 LLM，而是精准提取最相关的证据片段，减少噪声、节省 Token。

---

#### 智能体四：SentimentScoringAgent（情感评分）

```
输入：{dimension_id: [evidence_sentences]}
输出：list[DimensionScore]，list[AuditEntry]
```

**LLM 路径**：
```
System Prompt：你是情感分析专家，根据证据对指定维度打 1–5 分
User Prompt：维度名称 + 编号证据列表
LLM 返回：{"score": 4.5, "sentiment": "positive", "reasoning": "..."}
```

**启发式兜底**：

```python
positive_words = ["好", "棒", "赞", "满意", "优秀", "完美", "推荐", "great", "excellent", "love"]
negative_words = ["差", "烂", "糟", "不好", "失望", "垃圾", "后悔", "bad", "terrible", "hate"]

# 计票决定情感
if pos_count > neg_count:   → score=4.0, sentiment="positive"
if neg_count > pos_count:   → score=2.0, sentiment="negative"
else:                       → score=3.0, sentiment="neutral"
```

### 5.5 ReAct 编排层（OrchestratorAgent + Tools）

**文件**：`src/agents/orchestrator_agent.py`、`src/agents/tools.py`

`OrchestratorAgent` 是系统在固定流水线之上新增的 **ReAct 风格智能编排层**，将项目从预设顺序的四段式流水线升级为真正的 Agent：LLM 自主决定调用哪个工具、以何种顺序调用，并维持多轮对话上下文。

#### 核心公共接口

```python
class OrchestratorAgent:
    def chat(self, user_message: str) -> str:
        """处理用户消息，返回中文自然语言回复；多轮历史自动维护"""

    def analyze(self, review_text: str, review_id: str | None = None) -> ReviewAnalysisResult:
        """委托底层 Pipeline 运行结构化分析，返回 ReviewAnalysisResult"""

    def reset(self) -> None:
        """清空对话历史，开始新会话"""

    @property
    def history(self) -> list[dict]:
        """当前对话历史的只读视图（不含 system prompt）"""
```

**典型使用示例**：
```python
agent = OrchestratorAgent(pipeline)

# 第一轮：分析评论
reply = agent.chat("这款手机质量非常好，物流也很快")
# → LLM 依次调用 recognize_product → detect_dimensions → retrieve_evidence → score_sentiment
# → 返回中文评分总结

# 第二轮：追问（无需重新分析，直接从历史回答）
reply = agent.chat("哪个维度评分最低？")

# 开始新会话
agent.reset()
```

#### ReAct 循环（`_react_loop`）

每次调用 `chat()` 时，`_react_loop` 方法会：

1. 构建消息列表：`system prompt + 对话历史`
2. 调用 LLM，携带 4 个工具定义（`tools=TOOLS, tool_choice="auto"`）
3. **若 LLM 返回 `tool_calls`**：序列化工具调用 → 用 `ToolExecutor` 执行 → 将结果追加到 `messages` → 继续下一轮
4. **若 LLM 返回纯文本**：作为最终回复返回
5. 超过 `max_iterations`（默认 10）时返回安全提示，防止死循环

```
Reason ──► Act（工具调用）──► Observe（工具结果）
  ▲                                    │
  └────────────── next iteration ◄─────┘
                    … until final answer
```

#### 工具定义（`tools.py`）

将 4 个专职智能体封装为 OpenAI function-calling 工具，LLM 可通过名称和参数描述自主选择：

| 工具名 | 对应智能体 | 必要参数 |
|--------|-----------|---------|
| `recognize_product` | `ProductRecognitionAgent` | `review_text` |
| `detect_dimensions` | `DimensionDetectionAgent` | `review_text` |
| `retrieve_evidence` | `EvidenceRetrievalAgent` | `review_text`, `dimension_ids` |
| `score_sentiment` | `SentimentScoringAgent` | `review_id`, `evidence_map` |

**ToolExecutor** 负责解析 LLM 返回的 JSON 参数并派发到对应智能体，统一错误处理，返回 JSON 可序列化的结果字典。

#### 双模式运行

| 模式 | 参数 | 行为 |
|------|------|------|
| LLM 模式 | `use_llm=True`（默认） | ReAct 循环，LLM 动态选择工具和顺序 |
| 启发式模式 | `use_llm=False` | 直接调用底层 Pipeline 关键词兜底路径，无需 API Key |

### 5.7 评论聚类层（DBSCAN + LLM）

**文件**：`src/clustering/review_cluster.py`

用于对大批量评论按维度进行自动聚类，发现用户关注点的模式。

#### 完整流程

```
1. 对所有评论文本生成嵌入向量（EmbeddingModel）
         ↓
2. DBSCAN 聚类（余弦距离）
   - eps=0.3（邻域半径），min_samples=2（最小点数）
   - 自动识别噪声点（label = -1）
         ↓
3. 对每个簇：
   a. 计算簇质心
   b. 选取离质心最近的 3 条评论作为代表性评论
   c. 计算平均情感分数（如有）
         ↓
4. LLM 摘要生成（每个簇）
   - 输入：最多 10 条样本评论
   - 输出：≤10 字的核心总结词
   - 兜底：统计最高频词汇（过滤停用词）
         ↓
5. 输出 ClusterResult
```

**DBSCAN 选型原因**：
- 不需要预设类别数 k（评论主题数量未知）
- 自动识别噪声/离群点（孤立评论不强制归类）
- 适合高维余弦空间的密度聚类

## 6. 核心数据流：一条评论如何被分析

以评论 `"手机质量很好，但是快递太慢了，客服也不理人"` 为例：

```
输入文本
  │
  ▼
[Stage 1] ProductRecognitionAgent
  → 向量化评论 → 检索商品库 → "智能手机 Pro Max" (similarity=0.82)
  → 写入 AuditEntry (stage="product_recognition")
  │
  ▼ ProductMatch(product_id="phone_001", similarity=0.82)
  │
  ▼
[Stage 2] DimensionDetectionAgent
  → LLM 提取 → {"detected_dimensions": ["quality", "delivery", "service"]}
  → 写入 AuditEntry (stage="dimension_detection")
  │
  ▼ ["quality", "delivery", "service"]
  │
  ▼
[Stage 3a] EvidenceRetrievalAgent
  → 句子切分: ["手机质量很好", "快递太慢了", "客服也不理人"]
  → 临时向量化存入 ChromaDB
  → quality 查询 → ["手机质量很好"]
  → delivery 查询 → ["快递太慢了"]
  → service  查询 → ["客服也不理人"]
  → 写入 3 条 AuditEntry (stage="evidence_retrieval")
  → 删除临时集合
  │
  ▼ {quality: ["手机质量很好"], delivery: ["快递太慢了"], service: ["客服也不理人"]}
  │
  ▼
[Stage 3b] SentimentScoringAgent
  → quality:  LLM 评分 → score=4.8, sentiment="positive"
  → delivery: LLM 评分 → score=1.5, sentiment="negative"
  → service:  LLM 评分 → score=1.8, sentiment="negative"
  → 写入 3 条 AuditEntry (stage="sentiment_scoring")
  │
  ▼
汇总
  → overall_score = (4.8 + 1.5 + 1.8) / 3 = 2.7
  → overall_sentiment = "neutral"（2.5 < 2.7 < 4.0）

输出 ReviewAnalysisResult：
  ├── product_match: "智能手机 Pro Max" (0.82)
  ├── detected_dimensions: ["quality", "delivery", "service"]
  ├── dimension_scores: [质量4.8✅, 物流1.5❌, 服务1.8❌]
  ├── overall_sentiment: "neutral"
  └── overall_score: 2.7
```

## 7. 关键设计决策

### 7.1 双路径容错（LLM + 兜底）

每个依赖 LLM 的智能体都有关键词/启发式兜底逻辑。好处：
- **开发/测试阶段**无需 API Key，设置 `use_llm=False` 即可运行全流程
- **API 限流或故障**时系统不崩溃，降级运行
- **测试套件完全离线**：使用 `MockEmbeddingModel`（SHA256 哈希）+ `use_llm=False`

### 7.2 句子级 RAG（而非全文注入）

EvidenceRetrievalAgent 先把评论切句，再按维度检索最相关句子，而不是把整条评论喂给 LLM。

**优势**：
- 降低 Token 消耗
- 减少无关内容对评分的干扰
- 每条评分都有精准的原文依据

### 7.3 临时向量集合（无状态证据检索）

为每条评论创建独立的临时 ChromaDB 集合 `review_{id}`，分析结束后立即删除。

**优势**：
- 避免跨评论数据污染
- 不产生磁盘垃圾数据
- 无需预先索引评论库

### 7.4 全量审计日志

每个智能体在每次决策时都写入 `AuditEntry`，包含输入、输出、检索邻居、相似度、推理原因。

**优势**：
- 完整可追溯性，支持回溯分析任何评论的任何决策步骤
- 支持后续模型调优（有标注数据）
- 符合可解释 AI 的要求

### 7.5 配置外置化

所有商品、维度、系统参数都在 YAML 中定义，运行时可热更新，无需修改代码。Streamlit 配置管理页面直接读写 YAML 文件，实现**无代码运维**。

### 7.6 ReAct 编排 + 多轮对话

`OrchestratorAgent` 将固定流水线包装为 ReAct 智能体，带来三项能力提升：
- **动态工具选择**：LLM 可根据评论内容自主决定工具调用顺序，而非硬编码四步序列
- **自我纠错**：工具返回空结果时，LLM 可决定重试或调整参数
- **多轮对话**：`_history` 列表跨 `chat()` 调用持久化，用户可追问分析结果而无需重新分析

`max_iterations=10` 安全上限防止 LLM 无限循环；`use_llm=False` 模式下降级为直接调用流水线，确保测试和离线场景均可正常运行。

```python
# conftest.py
class MockEmbeddingModel:
    def encode(self, texts, normalize=True):
        vectors = []
        for text in texts:
            hash_bytes = hashlib.sha256(text.encode()).digest()
            rng = np.random.default_rng(int.from_bytes(hash_bytes[:8], 'big'))
            vec = rng.standard_normal(64)  # 64 维伪向量
            if normalize:
                vec /= np.linalg.norm(vec)
            vectors.append(vec)
        return np.array(vectors)
```

#### 模式 B：智能对话（OrchestratorAgent 模式）

用户直接在对话框输入，与 **OrchestratorAgent（ReAct 编排智能体）** 进行多轮对话。

**用户输入（第一轮）：**
> "这款手机质量非常好，物流也很快"

**Agent 内部 ReAct 循环（用户不可见）：**

```
用户消息 → LLM（GPT-4o-mini）推理
    → 调用 recognize_product  → ProductRecognitionAgent 识别商品
    → 调用 detect_dimensions  → DimensionDetectionAgent 检测维度
    → 调用 retrieve_evidence  → EvidenceRetrievalAgent 检索证据
    → 调用 score_sentiment    → SentimentScoringAgent 评分
    → LLM 无需再调用工具，生成中文自然语言总结
    → 返回最终回复（最大迭代 10 次）
```

**Agent 返回给用户（中文自然语言）：**

```

• 识别商品：iPhone 15（Apple，智能手机，相似度 82.10%）

各维度评分：
  😊 产品质量：4.5/5 (positive)
     ↳ 评论中明确提到质量很好，情感积极
  😊 物流配送：4.8/5 (positive)
     ↳ 提到物流快，用户满意

整体评价：positive（4.65/5）
```

**用户追问（第二轮，无需重新分析）：**
> "为什么物流评分这么高？"

Agent 直接从对话历史和已有工具调用结果回答，不重新运行分析流程，响应迅速。

> 本章从工程实现视角，系统梳理 MultiMind-Review 所采用的 Agent 设计范式、各层次使用的库与 SDK，以及关键技术决策的依据。

---

### 12.1 Agent 设计范式

#### 12.1.1 范式选型：Multi-Agent + ReAct

系统同时采用两种互补的 Agent 范式：

| 范式 | 对应实现 | 适用场景 |
|------|---------|---------|
| **专职 Agent 流水线（Pipeline Agents）** | `ReviewAnalysisPipeline` 顺序调用 4 个专职 Agent | 批量结构化分析、需要可预测输出时 |
| **ReAct（Reason-Act-Observe）** | `OrchestratorAgent` / `DatasetOrchestratorAgent` | 多轮对话、动态工具选择、用户追问 |

ReAct 范式的三步循环：

```
Reason（推理）
    LLM 读取 System Prompt + 对话历史 + 工具定义，决定下一步行动

Act（行动）
    LLM 输出 tool_calls → ToolExecutor 调用对应专职 Agent

Observe（观察）
    工具结果以 {"role": "tool"} 消息追加到上下文

    ↑──────── 循环至 LLM 输出纯文本回复（无 tool_calls）────────┘
```

安全上限：`max_iterations=10`，防止死循环；超限时返回友好提示。

#### 12.1.2 四专职 Agent 职责边界

```
┌─────────────────────────────────────────────────────┐
│              ReviewAnalysisPipeline                  │
│                                                     │
│  ①ProductRecognitionAgent  ──  向量库检索（RAG）      │
│  ②DimensionDetectionAgent  ──  LLM 语义理解          │
│  ③EvidenceRetrievalAgent   ──  句子级 RAG            │
│  ④SentimentScoringAgent    ──  LLM 推理 + 评分       │
│                                                     │
│  每个 Agent：单一职责 | LLM路径 + 兜底路径 | AuditEntry │
└─────────────────────────────────────────────────────┘
```

#### 12.1.3 工具调用机制（OpenAI Function Calling）

Agent 工具通过 OpenAI 的 Function Calling 协议暴露给 LLM，工具定义格式：

```python
{
    "type": "function",
    "function": {
        "name": "detect_dimensions",
        "description": "检测评论中涉及的评价维度...",
        "parameters": {
            "type": "object",
            "properties": {
                "review_text": {"type": "string", "description": "用户评论原文"}
            },
            "required": ["review_text"]
        }
    }
}
```

LLM 返回 `tool_calls` 字段时，`ToolExecutor.execute()` 负责：
1. 解析 JSON 参数
2. 派发到对应专职 Agent 方法
3. 统一 `try/except` 包装，返回 `{"error": str(exc)}` 而非抛出异常

### 12.2 Agent 层依赖库详解

#### 12.2.1 LLM 调用层

**库：`openai >= 1.0.0`**

  ```python
  from openai import OpenAI
  ```

#### 12.2.2 文本嵌入层

**库：`sentence-transformers >= 3.0.0`**

- **模型**：`sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`
- **关键接口**：
  ```python
  from sentence_transformers import SentenceTransformer
  model = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
  embeddings = model.encode(texts, normalize_embeddings=True)  # → shape (N, 384)
  ```
- **选型原因**：
  - 384 维向量，模型体积小（约 120 MB），推理速度快
  - 原生支持 50+ 语言，中英文评论无需分语言处理
  - `normalize_embeddings=True` 后向量点积 = 余弦相似度，直接与 ChromaDB 配合使用

**离线降级机制**（`src/rag/embeddings.py`）：

```python
# 若 HuggingFace 不可访问，自动切换 SHA256 哈希伪嵌入
def _hash_vec(self, text: str) -> np.ndarray:
    rng = np.random.default_rng(
        seed=int(hashlib.sha256(text.encode()).hexdigest(), 16) % (2**31)
    )
    return rng.standard_normal(128).astype(np.float32)  # 128 维确定性向量
```

此设计确保测试套件完全离线运行，`MockEmbeddingModel`（`tests/conftest.py`）同样采用此原理。

**环境变量配置**：

| 变量 | 默认值 |
|------|--------|
| `EMBEDDING_MODEL` | `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` |

#### 12.2.3 向量数据库层

**库：`chromadb >= 0.5.0`**

- **关键接口**：
  ```python
  import chromadb
  client = chromadb.PersistentClient(path="./chroma_db")
  collection = client.get_or_create_collection(
      name="products",
      metadata={"hnsw:space": "cosine"}  # 使用余弦距离
  )
  collection.upsert(ids=[...], embeddings=[...], documents=[...], metadatas=[...])

  ```
- **两类集合的生命周期**：

  | 集合名 | 生命周期 | 用途 |
  |--------|---------|------|
  | `products` | 持久化（跨会话） | 商品知识库，`ProductRecognitionAgent` 检索 |
  | `review_{uuid}` | 单次请求（分析后删除） | 评论句子临时索引，`EvidenceRetrievalAgent` 检索 |

- **选型原因**：
  - 本地持久化，无需单独部署向量数据库服务
  - 原生 HNSW 索引，余弦相似度高效检索
  - Python API 简洁，支持元数据过滤
  - `similarity = 1 - cosine_distance` 的计算封装在 `VectorStore.query()` 内

#### 12.2.5 聚类分析层

**库：`scikit-learn >= 1.4.0`**

- **关键接口**：
  ```python
  from sklearn.cluster import DBSCAN
  
  dbscan = DBSCAN(
      eps=0.3,           # 邻域半径（余弦距离空间）
      min_samples=2,     # 形成核心点的最少邻居数
      metric="cosine",   # 使用余弦距离
  )
  labels = dbscan.fit_predict(embeddings)  # -1 表示噪声点
  ```
- **选型原因**：
  - DBSCAN 不需预设类别数 k，适合主题数量未知的评论聚类
  - 自动识别噪声/离群点（不强制归类），更贴近真实场景
  - `metric="cosine"` 与嵌入向量的余弦空间一致

**库：`numpy >= 1.26.0`**

### 12.4 Agent 通信协议

各 Agent 之间通过**强类型 Pydantic 对象**传递数据，而非原始字符串，保证类型安全：

```
用户输入 (str)
    │
    ▼
ProductRecognitionAgent → (ProductMatch | None, AuditEntry)
    │
    ▼
DimensionDetectionAgent → (list[str], AuditEntry)   # 维度 ID 列表
    │
    ▼
EvidenceRetrievalAgent  → (dict[str, list[str]], list[AuditEntry])   # evidence_map
    │
    ▼
SentimentScoringAgent   → (list[DimensionScore], list[AuditEntry])
    │
    ▼
ReviewAnalysisPipeline  → ReviewAnalysisResult    # 聚合所有输出
```

在 ReAct 路径（OrchestratorAgent）中，Agent 输出通过 `ToolExecutor` 转换为 `dict`（JSON 可序列化），作为 `{"role": "tool"}` 消息追加至 LLM 上下文，实现 Observe 步骤。

### 12.5 双路径容错架构

所有依赖 LLM 的 Agent 均实现 **LLM 路径 + 启发式兜底** 的双路径设计：

```
use_llm=True  ─► LLM 路径（需要 OPENAI_API_KEY）
                   • DimensionDetectionAgent: 语义理解 → JSON 解析
                   • SentimentScoringAgent:   情感推理 → 1-5 评分
                   • OrchestratorAgent:       ReAct 工具调用

use_llm=False ─► 启发式路径（完全离线）
                   • DimensionDetectionAgent: 关键词命中匹配
                   • SentimentScoringAgent:   正/负词汇计票
                   • OrchestratorAgent:       直接调用 Pipeline
```

ProductRecognitionAgent 和 EvidenceRetrievalAgent 无需 LLM，仅依赖向量相似度（`sentence-transformers` + `chromadb`），在 `use_llm=False` 时同样完整运行。

### 12.6 测试中的 Agent 依赖隔离

**文件**：`tests/conftest.py`

测试套件通过依赖注入实现对外部服务的完全隔离：

| 外部依赖 | 隔离方式 |
|---------|---------|
| HuggingFace 嵌入模型 | `MockEmbeddingModel`（SHA256 哈希伪嵌入，64 维，无网络请求） |
| OpenAI LLM API | `use_llm=False`（Pipeline 和 OrchestratorAgent 均支持） |
| ChromaDB 持久化 | 测试使用临时目录（`tmp_path` fixture），测试结束自动清理 |

```python
# conftest.py — 核心 Mock
class MockEmbeddingModel:
    """确定性哈希嵌入，测试专用，无需网络"""
    def encode(self, texts, normalize=True):
        vectors = []
        for text in texts:
            seed = int.from_bytes(hashlib.sha256(text.encode()).digest()[:8], 'big')
            vec = np.random.default_rng(seed).standard_normal(64)
            if normalize:
                vec /= np.linalg.norm(vec)
            vectors.append(vec.astype(np.float32))
        return np.array(vectors)
```
